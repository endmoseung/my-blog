> 얼마 전 팀에 공유하려고 발표 자료를 하나 만들었습니다. AI로 UI를 뽑는 워크플로우를 열 장 남짓으로 정리했는데, 만들고 보니 슬라이드로는 다 못 담은 얘기가 꽤 남더라고요. 그 뒷얘기까지 붙여서 글로 남깁니다.

AI한테 화면을 만들어달라고 해보신 적 있나요. 요즘 모델들은 정말 잘 만듭니다. 문법에 맞는 코드가 나오고, 컴포넌트도 그럴듯하게 쪼개져 있고, 돌려보면 화면도 뜹니다. 그런데 이상하게 마음에 안 들어요.

어디서 본 듯한 회색 카드가 무더기로 나오고, 디자인 시스템에 없는 색을 그 자리에서 발명하고, hover나 빈 목록, 에러 같은 상태는 통째로 빠져 있습니다. 한 마디로 그럴듯한데 우리 제품 같지 않은 결과물이 나옵니다.

저는 회사에서 채용 관리 서비스를 만들면서 UI 작업 대부분을 AI와 함께 하고 있습니다. 처음엔 이게 AI의 한계인가 싶었는데, 지금은 생각이 다릅니다. AI가 못해서가 아니라 제가 두 가지를 안 줘서였어요. 하나는 제 취향을 전달할 방법, 다른 하나는 나온 결과를 판정할 기준입니다.

## 만들기 네 단계

이 두 가지를 채우려고 워크플로우를 만들었습니다. 만들기는 네 단계입니다.

```
① 레퍼런스 수집 → ② 디자인 시스템을 제약으로 → ③ 시안 HTML → ④ 이미지 + 스펙 → 구현
```

### 레퍼런스부터 모읍니다

말로 "깔끔하게, 모던하게"라고 해봐야 AI는 인터넷 평균값을 그립니다. 그게 바로 그 회색 카드 무더기죠. 그래서 만들기 전에 잘 만든 화면부터 모읍니다.

여기서 레퍼런스는 베끼려고 모으는 게 아닙니다. 제 취향을 말 대신 이미지로 전달하는 언어에 가까워요. "이런 느낌으로"라는 문장 한 줄보다 잘 고른 스크린샷 석 장이 훨씬 정확하게 전달됩니다. 저는 사내 레퍼런스 저장소를 따로 만들어서 좋은 화면을 계속 쌓아두고, 경쟁 제품 화면도 자동으로 수집해서 필요할 때 꺼내 씁니다.

### 디자인 시스템은 부탁이 아니라 제약으로

프롬프트에 "디자인 시스템 지켜줘"라고 쓰는 건 부탁입니다. 그리고 부탁은 지켜지지 않더라고요.

그래서 제약으로 바꿨습니다. 디자인 시스템의 토큰과 컴포넌트를 MCP 서버(AI가 도구를 직접 호출할 수 있게 하는 연결 규격)로 만들어서, AI가 우리 시스템을 검색해 실제 토큰을 그대로 쓰게 합니다. 여기에 "이 상황엔 이 UI를 쓴다"는 매핑 사전을 문서로 주고요.

쓰라는 것만큼 중요한 게 금지 목록입니다. 새 색 발명 금지, 디자인 시스템 밖 컴포넌트 금지. 이 선을 그어두면 결과물이 비로소 우리 제품처럼 보이기 시작합니다.

### 시안은 코드 밖에서, 여러 개를

코드베이스에 바로 구현하면 어떻게 되냐면, 마음에 안 들어도 버리기 아까워서 못 버립니다. 어중간한 걸 계속 고치게 돼요.

그래서 시안은 파일 하나짜리 HTML로 코드 밖에서 먼저 뽑습니다. 이건 쌉니다. 마음껏 만들고 마음껏 버릴 수 있어요. 그리고 한 방에 하나 만들지 않고 변형을 두세 개 나란히 뽑아서 비교합니다. 하나만 보면 그게 좋은지 나쁜지 판단이 안 서거든요. 둘을 나란히 놓으면 그제야 보입니다.

### 구현의 입력은 이미지와 스펙을 한 쌍으로

시안이 정해지면 그 이미지를 구현하는 에이전트한테 입력으로 넘깁니다. 요즘 모델들은 이미지를 보고 꽤 정확하게 그대로 만들어냅니다.

다만 이미지만 주면 반쪽입니다. hover했을 때, 목록이 비었을 때, 에러가 났을 때 어떻게 되는지는 정지 화면에 안 담겨 있으니까요. 그래서 동작과 상태를 적은 스펙 문서를 항상 같이 넘깁니다. 이미지는 어떻게 생겼는지를, 스펙은 어떻게 움직이는지를 맡는 거죠.

## 그런데 품질은 여기서 안 갈렸습니다

솔직히 고백하면, 저는 한동안 위의 네 단계만 팠습니다. 레퍼런스를 더 잘 고르고 프롬프트를 더 정교하게 다듬으면 결과가 좋아질 거라고요. 실제로 좋아지긴 했는데, 어느 선에서 멈추더라고요. 여전히 어딘가 어긋난 결과물이 통과되고 있었습니다.

돌아보니 문제는 만들기가 아니라 그다음이었습니다. 만든 걸 누가, 어떻게 판정하고 있었나. 아무도 안 하고 있었어요. 정확히는 만든 AI 본인한테 묻고 있었습니다.

### 만든 놈이 심사까지 보게 하지 않는다

시안을 만든 그 세션에다 "어때?"라고 물으면 답은 정해져 있습니다. "좋습니다." AI는 자기가 만든 걸 좋다고 합니다. 같은 대화 안에서는 더 심해요. 자기가 방금 내린 결정을 뒤집을 이유가 없거든요.

그래서 심사를 분리했습니다. 시안 변형들이 나오면, 만든 세션이 아니라 별도의 심사자가 채점합니다. 저희 경우엔 실제 사용자에 가까운 페르소나(채용 담당자 역할을 부여한 에이전트)한테 "당신이라면 어느 안이 편하겠냐"를 묻고, 이긴 안 하나만 결정본으로 남깁니다.

여기서 원칙이 하나 생겼습니다. 사람을 1차 검수자로 쓰지 않는다는 겁니다. AI가 만든 걸 곧바로 사람 눈앞에 가져오면, 사람이 매번 "이거 아닌데"를 반복하는 검수 기계가 됩니다. 판정을 먼저 통과한 것만 사람 앞에 올리고, 사람은 최종 결정만 합니다.

### 코드 diff 말고 화면을 판정한다

구현 쪽도 마찬가지입니다. AI가 "구현했습니다"라고 말하는 것, 그리고 코드 diff가 그럴듯해 보이는 것. 둘 다 UI가 잘 나왔다는 증거가 아닙니다. UI의 증거는 화면뿐이에요.

그래서 구현이 끝나면 Playwright(브라우저를 코드로 조작하는 도구)로 실제 렌더된 화면을 캡쳐해서, 결정본 시안과 나란히 놓고 대조합니다. 어긋나면 고치고 다시 캡쳐하고요. 이 판정을 통과하기 전엔 "완료"라는 말을 못 쓰게 해뒀습니다.

그러니까 전체 그림은 이렇게 됩니다.

```
레퍼런스 → DS 제약 → 시안 → [판정: 채점·결정] → 이미지+스펙 → 구현 → [판정: 화면 대조]
```

만들기 네 단계 사이에 판정이 두 번 들어가는 구조입니다. 제 경험상 품질이 갈린 건 앞의 네 단계가 아니라 이 두 번의 판정이었습니다.

## 이 루프를 사람 손으로 돌리지 않기

판정 루프가 좋다는 걸 알아도, 매번 사람이 수동으로 돌리면 그것대로 일이 됩니다. 바쁘면 건너뛰게 되고, 건너뛰는 순간 도로 원점이죠. 그래서 루프 자체를 하네스(harness, 작업이 정해진 절차를 벗어나지 못하게 잡아주는 장치)로 만들어 둡니다.

거창한 건 아니고 네 가지입니다.

Lint, Prettier, CI, git hook 같은 품질 게이트가 기본선을 기계로 지킵니다. 사람이 "세미콜론 빠졌어요"를 말하는 순간 지는 겁니다.

워크플로우를 고정합니다. 계획, 구현, 검증의 순서를 정해두고 검증을 통과해야 다음으로 넘어가게요. "빨리 해줘"라고 해도 순서는 못 건너뜁니다.

코드량이 큰 작업은 계획 단계에서 미리 쪼갭니다. AI는 큰 덩어리를 한 번에 삼키면 중간에 방향을 잃는 경우가 많아서, 작업 단위를 잘게 나눠두는 것만으로 실패율이 눈에 띄게 줄었습니다.

마지막으로 PR마다 AI가 1차 코드리뷰를 답니다. 사람은 그 위에서 판단만 하고요.

## 남은 생각

이 워크플로우를 몇 달 돌리면서 확신하게 된 게 하나 있습니다. 생성 능력은 제가 애쓰지 않아도 모델이 좋아질수록 알아서 올라갑니다. 실제로 작년에 공들여 다듬던 생성 프롬프트 중 상당수가 지금은 필요 없어졌어요. 그런데 판정은 다릅니다. 무엇이 우리 제품답고, 어떤 상태까지 챙겨야 완성이고, 어느 선을 넘으면 반려인지. 이건 모델이 아무리 좋아져도 저희가 정의해줘야 하는 영역으로 남아 있습니다.

그래서 요즘 누가 "AI로 UI 어떻게 잘 뽑아요?"라고 물으면 프롬프트 얘기 대신 이걸 묻게 됩니다. 만든 결과물을 지금 누가 판정하고 있나요? 만든 본인한테 "어때?"라고 묻고 있다면, 아마 거기가 병목일 겁니다.

물론 이것도 정답이라기보단 지금 시점의 제 방식입니다. AI도 빠르게 바뀌고 저도 계속 고쳐가는 중이라 몇 달 뒤엔 다른 얘기를 하고 있을지도 몰라요. 더 나은 방법을 쓰고 계시다면 저한테도 알려주세요.
